\section{Why This Document Exists}

We ran 20 AI agents writing Spring Boot tests.
We varied the prompt, the knowledge base, and the execution structure.
We measured everything.

The raw result is easy to state:
some configurations take twice as many steps and cost twice as much.
A bar chart shows that.

But the interesting questions are not in the bar chart.
\emph{Why} does one configuration take 69 steps and another take 96?
\emph{Where} is the time going?
\emph{Can we predict} what a new configuration will cost before running it?
\emph{What should we fix first}?

Those questions need a framework.
This document builds that framework from scratch,
using real numbers from our data,
and checks every claim against what we actually observed.

\subsection*{The stark contrast that motivates everything}

Two runs from the same experiment, same model, same task:

\begin{center}
\renewcommand{\arraystretch}{1.3}
\begin{tabular}{@{}lcccc@{}}
\toprule
& \textbf{Steps} & \textbf{BUILD calls} & \textbf{FIX calls} & \textbf{Cost} \\
\midrule
\textbf{hardened+skills+sae} (run 3, r17) & 69 & 8 & 2 & \$3.49 \\
\textbf{hardened+skills+sae+forge} (run 2, r14) & 96 & \textbf{22} & 8 & \textbf{\$5.67} \\
\bottomrule
\end{tabular}
\end{center}

Same model. Same codebase. Same task. One agent ran the build 22 times.
The other ran it 8 times and finished much more cleanly.
The spiral run cost 63\% more and took 43 minutes instead of 26.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{../figures/cost-steps-comparison.png}
\caption{Mean cost (USD) and mean step count per variant across all runs.
         Red bar = the spiral variant (forge). Blue = all others.
         Error bars show standard deviation across N=3 runs per variant.
         The forge variant has the highest variance --- it can be the best or worst run
         depending on whether the agent hits a rework spiral.}
\end{figure}

Why? What happened inside that agent during those 22 build attempts?
That is what this document is trying to explain.

\section{Step 1 --- Label Every Move (The 8 States)}

An AI agent makes tool calls.
Each call is a concrete action: read a file, run the build, write a test, edit a failure.
The first thing we do is \textbf{group these into 8 semantic categories} --- states.

\begin{table}[H]
\centering
\begin{tabular}{@{}l p{4cm} p{6.5cm}@{}}
\toprule
\textbf{State} & \textbf{What is happening} & \textbf{What tool calls look like} \\
\midrule
EXPLORE     & Navigating the codebase          & Read, Glob, Grep on source files \\
READ\_KB    & Consulting the knowledge base    & Read on \texttt{knowledge/} directory \\
READ\_SKILL & Loading a skills package         & Skill tool invocation \\
JAR\_INSPECT& Searching Maven cache for imports& Bash: \texttt{jar tf \textasciitilde/.m2/...} \\
WRITE       & Writing test files               & Write on \texttt{src/test/} \\
BUILD       & Running the test suite           & Bash: \texttt{./mvnw clean test} \\
VERIFY      & Checking coverage results        & Read on JaCoCo report \\
FIX         & Editing tests after failure      & Edit on \texttt{src/test/} \\
\bottomrule
\end{tabular}
\caption{The 8 states. Every tool call maps to exactly one.}
\end{table}

Each run is now a sequence of labels:
\begin{codebox}
\texttt{EXPLORE, EXPLORE, READ\_KB, EXPLORE, WRITE, WRITE, BUILD, FIX, BUILD, VERIFY}
\end{codebox}

That 69-step run from above? Mostly EXPLORE and WRITE, with one BUILD→FIX→BUILD.
The 96-step spiral? Long stretches of BUILD→FIX→BUILD→FIX→BUILD$\ldots$

\section{Step 2 --- The Heatmap: Reading Agent Behavior at a Glance}

Once we have labeled sequences, we count transitions:
of all the times the agent was in state X, what fraction of the time did it go to state Y?

\[
P(i, j) = \frac{\text{times agent went from state } i \text{ to state } j}
               {\text{total times agent left state } i}
\]

This is the \textbf{transition probability matrix}, visualized as a heatmap.

\begin{keybox}
\textbf{How to read the heatmap:}
\vspace{4pt}
\begin{itemize}[leftmargin=1.5em,itemsep=2pt]
  \item \textbf{Rows (Y-axis)}: where the agent is \emph{right now}
  \item \textbf{Columns (X-axis)}: where it goes \emph{next}
  \item \textbf{Bright diagonal cell}: the agent stays in the same state (repetition)
  \item \textbf{Bright off-diagonal cell}: a mode switch
  \item Each row sums to 1.0 --- the agent must go somewhere
\end{itemize}
\end{keybox}

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{../figures/transition-probability-matrix.png}
\caption{Overview: transition probability matrices for all 7 variants.
         Yellow = low probability, orange = medium, dark red = high.
         A bright diagonal cell means the agent \emph{stays} in that state.
         A bright off-diagonal cell means a mode switch.
         The full-size per-variant heatmaps in the following sections
         show each cluster in detail.}
\end{figure}

\subsection*{Reading the story}

Common off-diagonal signals and what they mean:

\begin{center}
\begin{tabular}{@{}l p{8cm}@{}}
\toprule
\textbf{Bright cell} & \textbf{What it tells you} \\
\midrule
WRITE $\to$ BUILD       & Agent finished writing, now running tests (good) \\
BUILD $\to$ FIX         & Tests failed, agent is editing (rework begins) \\
FIX $\to$ BUILD         & Tried again after edits (rework continuing) \\
BUILD $\to$ VERIFY      & Build passed, checking coverage (good) \\
EXPLORE $\to$ JAR\_INSPECT & Agent hit import gap, searching Maven cache (friction) \\
JAR\_INSPECT $\to$ EXPLORE & Gave up on JAR, back to browsing (escaped) \\
\bottomrule
\end{tabular}
\end{center}

Compare any two variants' heatmaps side by side:
the differences in bright cells tell you exactly what behavior changed.

\section{Step 3 --- The Three Clusters: Naming What the Heatmap Shows}

The heatmap has structure, but structure without vocabulary is hard to discuss.
\textbf{Clusters} name the groups of states that form feedback loops ---
states the agent bounces between before moving forward.

We see three clusters in our data.

\subsection{The JAR\_INSPECT Cluster --- Friction, Fully Addressable}

\textbf{What it is}:
The agent hits an import error in a test it's writing.
It doesn't know which package the class belongs to.
There's no source file to check.
So it does the programmer's last resort: decompile JARs from the
\texttt{\textasciitilde/.m2} Maven cache, reading binary class files to find the class name.

\textbf{What the heatmap shows}:
A high diagonal on JAR\_INSPECT means the agent inspects one JAR, finds a clue,
then inspects another JAR. It stays in the cluster.
Look at the \texttt{hardened} heatmap below: the JAR\_INSPECT row has a strong red cell
on its own diagonal --- once in this cluster, the agent keeps searching.
Compare to \texttt{hardened+skills}: that whole row is pale --- the agent never enters it.

\begin{figure}[H]
\centering
\begin{minipage}[t]{0.48\textwidth}
  \includegraphics[width=\textwidth]{../figures/heatmap-hardened.png}
  \caption*{\textbf{hardened}: JAR\_INSPECT diagonal is bright red (high self-loop).
            11\% of all steps are JAR inspection.}
\end{minipage}\hfill
\begin{minipage}[t]{0.48\textwidth}
  \includegraphics[width=\textwidth]{../figures/heatmap-hardened-skills.png}
  \caption*{\textbf{hardened+skills}: JAR\_INSPECT row is nearly white.
            Skills provide the import directly --- no searching needed (1\%).}
\end{minipage}

         The diagonal self-loop (agent stays in JAR\_INSPECT) disappears completely
         when the knowledge base answers the import question directly.}
\end{figure}

\textbf{The paradox}:
Adding a hardened prompt \emph{increases} JAR searching (11\% vs 7\% for simple).
Why? The hardened prompt makes the agent \emph{thorough} about finding imports
without giving it the knowledge of \emph{where} to look.
Thoroughness without direction = maximum searching.

\textbf{The fix}:
The skills package drops JAR\_INSPECT to 1\%.
It provides the correct import directly.
The agent never needs to search.

\begin{center}
\begin{tabular}{@{}lrl@{}}
\toprule
\textbf{Variant} & \textbf{JAR \%} & \textbf{Why} \\
\midrule
simple                  & 7.9\%  & Baseline, hits some import gaps \\
hardened                & 11.0\% & Thorough without direction: maximum search \\
hardened+kb             & 10.1\% & KB reduces gaps but doesn't eliminate them \\
hardened+skills         & 1.0\%  & Skills provides the answer: no search needed \\
hardened+skills+sae     & 7.1\%  & pre-analysis trades some JAR reduction for step efficiency \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Why this is good news}:
Every percentage point in JAR\_INSPECT is recoverable.
It represents time the agent spends searching for information the knowledge base can provide.
This cluster exists because of a knowledge gap, and knowledge gaps are fixable.

\subsection{The FIX/BUILD Rework Loop --- Overhead With a Measurable Cost}

\textbf{What it is}:
The agent writes tests. BUILD fails. Agent edits (FIX). BUILD again. Maybe fails again.
Edit again. The loop continues until the tests compile and pass.

\textbf{The spiral case}:
In our data, \texttt{hardened+skills+sae+forge} run 2 (r14) is the clearest spiral:

\begin{codebox}
96 total steps --- 22 BUILD calls --- 8 FIX calls --- 6 FIX$\to$BUILD cycles
Duration: 43 minutes --- Cost: \$5.67
\end{codebox}

Compare to the cleanest run in the dataset (\texttt{hardened+skills+sae}, run 3, r17):

\begin{codebox}
69 total steps --- 8 BUILD calls --- 2 FIX calls --- 1 FIX$\to$BUILD cycle
Duration: 26 minutes --- Cost: \$3.49
\end{codebox}

22 builds versus 8. \$5.67 versus \$3.49. 43 minutes versus 26.
The forge two-phase design (deep exploration first, then a sustained act phase)
seems to produce more thorough initial test writing ---
but when that writing hits a compilation problem, the agent retries aggressively.

The FIX/BUILD loop appears as mutual high probability between BUILD and FIX rows.

\subsection{The PRODUCTIVE Flow --- What We Want More Of}

\textbf{What it is}:
EXPLORE $\to$ WRITE $\to$ BUILD $\to$ VERIFY.
Forward progress: understand the task, write tests, they pass, check coverage, done.

\textbf{What changes it}:
Structured execution (pre-analysis) front-loads a deep exploration phase.
The agent arrives at WRITE already knowing what to write.
Each WRITE step is more targeted.
Fewer FIX/BUILD loops follow.

The step-count difference between simple (109 steps) and hardened+skills+sae (75 steps)
is not abstract --- it is specific transitions that shifted.

The Sankey diagram makes the difference visceral.

\section{Step 4 --- The Markov Property: Why This Math Works}

We've been computing probabilities from counts.
That is valid only if a particular assumption holds.

\begin{keybox}
\textbf{The Markov Property (First-Order):} the next state depends only on the
\emph{current} state --- not on the full history of how the agent got there.

If the agent is in BUILD, the probability of going to FIX next is the same
whether it arrived at BUILD from WRITE (first attempt) or from FIX (a retry).
\end{keybox}

A sequence of states obeying this rule is called a \textbf{Markov chain},
named after Andrey Markov who worked out the mathematics in 1906.
The name is intimidating; the idea is simple: one step of history is enough.

\begin{analogybox}
\textbf{Board game analogy}:
The agent is a player on a board. Each square is a state.
At each turn, you roll dice and move.
Where you go depends only on where you are now ---
not on the path you took to get there.

This is a Markov chain.
The math for board games (how long to win? which squares do you visit most?)
was worked out decades ago. We're applying it to agents.
\end{analogybox}

\subsection*{First-Order vs Second-Order Markov: The Intuition}

\textbf{First-order}: what you do next depends only on what you're doing now.
The heatmap encodes first-order: one row per current state.

\textbf{Second-order}: what you do next depends on your \emph{last two} actions.
Second-order would need a bigger heatmap: not 8 rows (current state)
but 64 rows (all 8$\times$8 combinations of previous + current state).

We tested whether first-order is adequate.
The answer is nuanced --- and this is where many people (including us initially)
get the intuition backwards.

\subsection*{Where First-Order Holds and Where It Doesn't}
\label{sec:honest}

\begin{warningbox}
\textbf{Common misconception}: ``Within clusters, first-order breaks down.''

\textbf{The truth}: it's the opposite.
\end{warningbox}

\textbf{Self-loops within clusters are the \emph{most} Markov behavior in the data.}

Once the agent is in JAR\_INSPECT, the probability of staying in JAR\_INSPECT
does not depend on how many JAR steps it has already done, or what it was doing
before it entered the cluster.
It's essentially memoryless.
The agent in a JAR loop is just ``stuck'' in a probabilistic sense ---
and that probability is the same every step.

KL divergence measures how much knowing the previous state changes the prediction.
Zero bits = perfectly Markov. The three self-loop transitions:
\begin{itemize}
  \item EXPLORE $\to$ EXPLORE: KL = 0.03 bits (essentially memoryless)
  \item WRITE $\to$ WRITE: KL = 0.01 bits (essentially memoryless)
  \item FIX $\to$ FIX: KL = 0.01 bits (essentially memoryless)
\end{itemize}

These three account for 64\% of all transitions.
The heatmap is accurate for nearly two-thirds of everything that happens.

\textbf{Where first-order breaks down: the cluster exits.}

The non-Markov behavior lives in the recovery transitions ---
what happens when the agent escapes a failure state back into EXPLORE:

\begin{center}
\begin{tabular}{@{}llrc@{}}
\toprule
\textbf{Previous state} & \textbf{Current state} & \textbf{KL (bits)} & \textbf{What it means} \\
\midrule
FIX     & EXPLORE & 2.10 & Where you came from before FIX changes your exploration \\
BUILD   & EXPLORE & 0.77 & Build failure vs build success leads to different exploration \\
VERIFY  & EXPLORE & 1.17 & Near-miss recovery is context-dependent \\
\midrule
EXPLORE & EXPLORE & 0.03 & Self-loop is memoryless \\
WRITE   & WRITE   & 0.01 & Self-loop is memoryless \\
\bottomrule
\end{tabular}
\end{center}

The FIX$\to$EXPLORE transition has 2.10 bits of second-order information.
That is enormous by information-theory standards.
Knowing whether the agent came to FIX from a first-attempt BUILD
or from its sixth BUILD attempt dramatically changes the prediction of what it explores next.

\textbf{Why this makes human sense}:

\begin{analogybox}
A developer who just fixed their first compile error explores the codebase
looking for what to test next --- open-ended, curious.

A developer who just spent six rounds fixing the same failing test
explores very differently --- targeted, frustrated, hypothesis-driven.
``Where did that import come from? Did I spell that class name right?
Let me check the package one more time.''

The \emph{content} of exploration changes based on context.
First-order Markov doesn't capture that.
But it doesn't need to for counting total steps.
\end{analogybox}

\subsection*{So Is First-Order Markov Adequate?}

\textbf{For predicting step counts: yes.}

Over a 100-step trajectory, the recovery transitions happen a handful of times.
Each introduces a small local error in the prediction.
Those errors cancel out.
Zero mean bias in leave-one-out cross-validation.

\textbf{For modeling individual decisions at cluster exits: no.}

The second-order signal at recovery transitions is real and large (2.10 bits).
If you want to understand \emph{why} one agent recovers cleanly and another spirals,
you need second-order (or richer) models.

\begin{analogybox}
\textbf{The chess analogy}:

A model that says ``each chess move is independent of the previous one''
will be wrong about individual decisions --- grandmasters absolutely consider the last
several moves. But that same model might still correctly predict that a typical game
lasts about 40 moves.

\emph{Trajectory length is robust to local violations of the Markov assumption.}

What our second-order test found: the agent's \emph{recovery behavior}
is context-dependent. This is interesting and interpretable.
But it doesn't break our step-count prediction.
\end{analogybox}

\section{Step 5 --- The Fundamental Matrix: Predicting Step Counts}
\label{sec:fundamental}

Now we use the mathematics that makes this more than just counting.

\subsection*{Absorbing States: The Game Has to End}

Our agent eventually finishes.
The build passes (success) or the run times out (failure).
These are \textbf{absorbing states}: once entered, never left.
Everything else (EXPLORE, WRITE, BUILD, FIX, \ldots) is \textbf{transient}:
states the agent visits but eventually exits.

The transition matrix splits:
\begin{itemize}
  \item $Q$ = transitions between transient states (the interesting part)
  \item $R$ = transitions from transient states to absorbing states (exit probabilities)
\end{itemize}

\subsection*{The Formula}

\begin{keybox}
\[
N = (I - Q)^{-1}
\]

$N(i, j)$ = expected number of times the agent visits state $j$,
starting from state $i$, before finishing.

Sum the first row (EXPLORE = starting state):
$\sum_j N(0, j)$ = expected total steps before the run ends.
\end{keybox}

This is a matrix inverse --- ordinary linear algebra, not exotic mathematics.
An 8$\times$8 matrix inverts in milliseconds.
The result is exact, not an approximation.

\subsection*{Validation: Does It Actually Work?}

We tested this with leave-one-out cross-validation.
\textbf{Procedure}: take 3 runs of a variant.
Hide 1 run (the held-out run).
Fit $P$ from the other 2.
Compute $N = (I-Q)^{-1}$.
Predict how many steps the hidden run will take.
Compare to what actually happened.

\begin{findingbox}
\textbf{Leave-one-out cross-validation results (5 variants with N=3):}
\vspace{6pt}
\begin{center}
\begin{tabular}{@{}lrrc@{}}
\toprule
\textbf{Variant} & \textbf{Actual steps/run} & \textbf{Predicted} & \textbf{MAE} \\
\midrule
hardened            & 100.7 & 100.7 & 19.7 \\
hardened+kb         &  82.3 &  82.3 &  8.3 \\
hardened+sae        &  79.7 &  79.7 & 13.3 \\
hardened+skills     & 101.0 & 101.0 & 11.0 \\
hardened+skills+sae &  74.7 &  74.7 & 10.3 \\
\bottomrule
\end{tabular}
\end{center}
\vspace{4pt}
Zero mean bias. MAE of 8--20 steps ($\approx \pm 15\%$).
\end{findingbox}

Predicted = actual across all five variants.
The fundamental matrix captures something real about how these agents work.

\subsection*{What the Matrix Decomposes}

Beyond the total step count, $N$ gives you per-state expected visits:

\begin{itemize}
  \item $N(0, \text{JAR\_INSPECT})$ = expected JAR searches per run
  \item $N(0, \text{FIX})$ = expected rework steps per run
  \item $N(0, \text{WRITE})$ = expected productive writes per run
  \item $N(0, \text{BUILD})$ = expected build attempts per run
\end{itemize}

For the spiral run (forge r14) versus the clean run (r17):
\begin{itemize}
  \item forge r14 contributes $N(0, \text{BUILD}) \approx 22$ to the expected value calculation
  \item r17 contributes $N(0, \text{BUILD}) \approx 8$
  \item The expected value for forge across 3 runs lands at $\approx 15$ expected BUILD steps
\end{itemize}

The matrix is not just a step-count oracle.
It is a \textbf{decomposition of where agent time goes}, per state.

\centering
\includegraphics[width=\textwidth]{../figures/fundamental-matrix-analysis.png}
\caption{Fundamental matrix: expected visits per state.
         Large diagonal entries are loops.
         The BUILD row for forge variants reflects rework.

\end{figure}

\section{Step 6 --- Can We Predict Before Running?}

\begin{keybox}
\textbf{Short answer: yes, in principle. With current data: with caveats.}
\end{keybox}

The fundamental matrix gives you step counts from a calibrated $P$ matrix.
If you're designing a new variant,
you can modify $P$ to model the expected behavior change and predict the result
before spending any money on agent runs.

\subsection*{How It Works in Practice}

Suppose you've run the \texttt{hardened} variant and have its calibrated $P$ matrix.
You're designing a new KB that will tell the agent exactly which imports to use.
You expect this to eliminate JAR\_INSPECT almost entirely.

\textbf{Step 1}: Take $P_{\text{hardened}}$.

\textbf{Step 2}: The JAR cluster occupies 11\% of steps in hardened.
In the transition matrix, this means $P(\text{EXPLORE}, \text{JAR\_INSPECT})$ is elevated.
Set those entries to near zero: the agent will no longer enter the JAR cluster.

\textbf{Step 3}: Redistribute that probability.
When an agent stops searching JARs, where does it go instead?
It goes to WRITE --- the KB provided the answer, the agent can proceed.
So increase $P(\text{EXPLORE}, \text{WRITE})$ by the amount removed from
$P(\text{EXPLORE}, \text{JAR\_INSPECT})$.

\textbf{Step 4}: Recompute $N = (I-Q)^{-1}$.

\textbf{Step 5}: Read the new expected step count. That's your prediction.

\subsection*{The Calibration Question}

This works well for \emph{targeted} interventions that affect a specific part of $P$.
The JAR cluster is ideal: we know exactly which transition to zero out.

It works less well for \emph{diffuse} interventions that might change many entries of $P$
in unpredictable ways. Adding a new structured execution prompt could shift dozens of
transition probabilities simultaneously.

The practical rule: \textbf{if you know which specific transitions your intervention
should change, you can forecast. If you're making a sweeping change, run the experiment.}

\subsection*{A Worked Example from Our Data}

The delta heatmap between hardened and hardened+skills shows:
\begin{itemize}
  \item $P(\text{EXPLORE}, \text{JAR\_INSPECT})$ dropped by approximately 0.08
  \item $P(\text{EXPLORE}, \text{WRITE})$ rose by approximately 0.05
\end{itemize}

If we had only run \texttt{hardened} and wanted to forecast \texttt{hardened+skills},
we'd apply those changes to $P_{\text{hardened}}$, recompute $N$,
and predict the new step count.
The actual step counts (hardened: 100.7, hardened+skills: 101.0) are nearly identical ---
because on petclinic, the skills package's main effect is quality, not efficiency.
The JAR reduction happened but other transitions compensated.

This points to the next experiment:
on a \emph{domain-specific codebase} where JAR searching is more prevalent,
the JAR reduction will translate directly to step savings,
and the forecast will be more useful.

\section{Step 7 --- What This Enables (Transition Probability Engineering)}

We have a name for the practice of deliberately modifying transition probabilities
to improve agent efficiency: \textbf{Transition Probability Engineering (TPE)}.

The workflow:
\begin{enumerate}
  \item \textbf{Measure}: run an agent, build its $P$ matrix
  \item \textbf{Diagnose}: which cluster is costing the most steps?
    (Look at which states have high $N(0, j)$ --- expected visit counts from the fundamental matrix.)
  \item \textbf{Design}: what intervention will shift the problematic transitions?
    (KB to answer the question the agent is searching for.
    Structured execution to route around the loop.
    Skills to provide patterns the agent is re-discovering.)
  \item \textbf{Forecast}: modify $P$, recompute $N$, predict the new step count
  \item \textbf{Run and measure}: compare actual to forecast --- update your model
  \item \textbf{Repeat}
\end{enumerate}

This turns agent improvement from ``try things and see'' into an engineering discipline
with predictive and diagnostic tools.

\subsection*{The diagnostic questions the math answers}

\begin{itemize}
  \item \textbf{Where is time going?}
    The fundamental matrix row $N(0, \cdot)$ gives expected visits per state.
    Sort descending: the top states are where the agent spends its time.

  \item \textbf{Which time is wasted?}
    JAR\_INSPECT is always waste. FIX/BUILD beyond 2--3 cycles is likely waste.
    EXPLORE beyond a certain threshold indicates the agent is lost.

  \item \textbf{What specifically changed between variants?}
    The delta heatmap $\Delta P = P_B - P_A$ shows which transitions shifted.
    This is \emph{causal attribution}, not just ``steps went down.''

  \item \textbf{What is the bottleneck?}
    The state with the highest expected visits relative to a clean run is the bottleneck.
    Fix that state first.
\end{itemize}

The step-count analysis gives us the \emph{efficiency} axis.
But we also measured \emph{quality}: how good were the tests the agent produced?

The T3 judge read each test suite and scored it on 6 criteria
(test slice selection, assertion quality, error coverage, domain patterns,
coverage target selection, version-aware patterns).
Score range: 0.0--1.0.

\textbf{Key findings}:

\begin{findingbox}
\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Prompt hardening is the biggest quality driver.}
    simple $\to$ hardened: $+$0.067 quality. The largest single jump.

  \item \textbf{pre-analysis drives efficiency at a quality cost.}
    hardened+sae: $-$27\% steps, but quality drops to 0.789 (back near simple's 0.783).

    On a novel codebase, this ratio should change.

\section{Step 9 --- What This Experiment Cannot Show (And What Comes Next)}

This is the section where we are honest.

\subsection*{The petclinic ceiling}

Spring-petclinic is one of the most documented Spring Boot applications on earth.
The model has seen it in training data.
Starting from 0\% coverage, all variants reach $\approx$92--94\% coverage.
Coverage is not a discriminator --- the model knows petclinic's domain.

This means:
\begin{itemize}
  \item The knowledge injection (KB, skills) can improve \emph{efficiency} (step count)
  \item It \emph{cannot} improve \emph{outcome quality} beyond the model's existing knowledge
  \item To prove the quality thesis, we need a domain the model doesn't know
\end{itemize}

\subsection*{What petclinic does prove}

``Petclinic is the negative control --- it proves the boundary condition.''

Structured execution (pre-analysis) always helps, even when the model knows the domain.
Knowledge injection helps efficiency even on known codebases.
The bigger claim --- that knowledge improves outcomes on novel domains ---
requires the next experiment.

\subsection*{Next experiments}

\begin{enumerate}
  \item \textbf{Domain-specific codebases with partial existing coverage}:
    OpenWMS (warehouse management), Apache Fineract (banking), or ThingsBoard (IoT).
    Pick repos at 40--70\% existing coverage.
    Task: improve from 65\% to 85\%, not write from scratch.
    Now the JAR cluster will be larger (the model doesn't know warehouse allocation rules)
    and the KB should show outcome improvement, not just efficiency improvement.

  \item \textbf{SWE-bench issue fix (fix/no-fix outcome)}:
    The pytest KB is already built (5,908 items).
    Run baseline and KB-augmented agents on 10--20 SWE-bench Lite issues.
    Compare fix rates AND Markov signatures.
    The key question: does the behavioral signature of a spiraling agent
    predict failure before the run ends?
    This is where second-order analysis becomes practically important.
\end{enumerate}

\section{The Honest Summary}

\begin{findingbox}
\textbf{What the Markov framework gives us:}

\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Cluster vocabulary} --- precise names for agent failure modes.
    JAR\_INSPECT (knowledge gap), FIX/BUILD loop (rework), PRODUCTIVE flow (forward progress).
    Without this, you say ``the agent was slow.''
    With this, you say ``the agent has a JAR cluster problem.''

  \item \textbf{Causal attribution} --- the delta heatmap shows which transitions
    changed between variants. ``Adding skills reduced EXPLORE$\to$JAR\_INSPECT by 10 points''
    is a stronger claim than ``step count dropped by X.''

  \item \textbf{Step-count prediction} --- zero mean bias in leave-one-out cross-validation.
    The fundamental matrix predicts how long a variant's runs take.

  \item \textbf{Pre-run forecasting} --- for targeted interventions
    (``I will eliminate the JAR cluster''), modify $P$ and predict the result.
    Currently speculative; becomes practical as we calibrate across more codebases.
\end{enumerate}

\vspace{6pt}
\textbf{What first-order Markov cannot claim:}
\vspace{4pt}
\begin{itemize}[leftmargin=1.5em]
  \item That it models individual transition decisions accurately (formally rejected, $p \approx 0$)
  \item That recovery transitions are memoryless (they aren't --- 2.10 bits at FIX$\to$EXPLORE)
  \item That the numbers generalize across codebases (untested)
\end{itemize}

\vspace{6pt}
\textbf{The honest publishable claim:}
\vspace{4pt}
\begin{quote}
``Agent tool-call traces exhibit measurable second-order dependence
(mean KL = 0.49 bits, $p < 0.001$), concentrated in recovery transitions
(FIX$\to$EXPLORE: KL = 2.10 bits). Despite this formal rejection of first-order Markov,
the fundamental matrix accurately predicts per-variant trajectory length:
leave-one-out cross-validation yields zero mean bias with MAE of 8--20 steps.
We use first-order Markov as an engineering approximation for step-count prediction,
acknowledging that recovery transitions exhibit genuine second-order structure.''
\end{quote}
\end{findingbox}

\section{Glossary}

\begin{description}[leftmargin=7em,labelindent=0em,itemsep=3pt]
  \item[State] A semantic label for one category of tool calls (EXPLORE, WRITE, BUILD\ldots)
  \item[Transition] The agent moving from one state to the next
  \item[$P$ matrix] Transition probability matrix; $P(i,j)$ = probability of going from $i$ to $j$
  \item[Heatmap] Visual of $P$; rows = current state, columns = next state
  \item[Markov chain] A sequence where each transition depends only on the current state
  \item[First-order] Next state depends on current state only
  \item[Second-order] Next state depends on current + previous state
  \item[KL divergence] Bits of extra information that knowing the previous state adds;
    zero = first-order is perfect
  \item[Absorbing state] A state the agent enters and never leaves (build success, timeout)
  \item[Transient state] A state the agent may visit but eventually leaves
  \item[$Q$ matrix] Sub-matrix of $P$: transient-to-transient transitions only
  \item[Fundamental matrix] $N = (I-Q)^{-1}$; $N(i,j)$ = expected visits to $j$ from $i$
  \item[Expected steps] $\sum_j N(0,j)$: total expected tool calls before the run ends
  \item[JAR cluster] EXPLORE$\leftrightarrow$JAR\_INSPECT loop; agent searching Maven cache
  \item[FIX/BUILD loop] BUILD$\leftrightarrow$FIX rework cycle; compile-debug loop
  \item[PRODUCTIVE flow] EXPLORE$\to$WRITE$\to$BUILD$\to$VERIFY; forward progress
  \item[Rework spiral] Extreme FIX/BUILD loop; forge r14 = 22 builds, 6 cycles
  \item[TPE] Transition Probability Engineering: deliberately modifying $P$ to improve agent efficiency
  \item[$\Delta P$] Delta heatmap ($P_B - P_A$); shows which transitions changed between variants
  \item[Leave-one-out CV] Validation method: fit on N$-$1 runs, predict the Nth, compare
  \item[MAE] Mean Absolute Error: average of $|\text{predicted} - \text{actual}|$ across folds
\end{description}
